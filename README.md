# codex-config

这个仓库用于整理我给 Codex / Agent 配置的协作规则、行为边界和运行方式，重点不是“放开代理”，而是让代理在受控前提下高效工作。

## 项目定位

这里记录的是一套面向日常开发协作的 Agent 限制方案，核心目标包括：

- 在执行动作前先说明意图
- 对高风险操作保留明确边界
- 减少对普通只读动作的重复确认
- 让沟通规则和运行时限制各司其职

当前仓库重点围绕 `v1` 与 `v2` 两套规则演进展开，其中 `v2` 更强调：

- 用运行时硬限制拦住真正危险的动作
- 用规则文件约束沟通方式和行为边界
- 避免把所有普通操作都拖入高频审批流程

## 核心内容

- `AGENTS.v1.md`：早期版本的规则约束
- `AGENTS.v2.md`：当前主推版本的规则设计
- `README.v1.md`：`v1` 方案说明
- `README.v2.md`：`v2` 方案说明
- `Codex.sh`：相关启动或辅助脚本
- `test/`：简单测试样例

## 设计思路

这套配置关注的不是“让 Agent 完全自动执行”，而是平衡三件事：

1. 让 Agent 在行动前先讲清楚自己要做什么
2. 让真正会改变状态的动作受到运行时限制
3. 让普通查看、分析、整理这类任务不要被反复打断

换句话说，这个仓库更像一个 Agent 协作策略实验场，而不是单纯的配置文件备份目录。

## 适用场景

适合以下需求：

- 希望 Agent 先解释再行动
- 希望文件修改、测试、脚本执行、Git 操作保持受控
- 希望减少“明明会被沙箱拦住，还要先确认一次”的重复流程

## 当前推荐入口

如果你第一次查看这个仓库，建议优先阅读：

1. `README.v2.md`
2. `AGENTS.v2.md`

它们基本代表了当前这套协作规则的主要方向。
