# codex-config

这个仓库用于记录我给 Codex/Agent 使用的限制方案。

## v2 限制方案

`v2` 的核心是把真正必要的边界收敛到两条硬限制上：

```toml
sandbox_mode = "read-only"
approval_policy = "on-request"
```

它们分别负责：

- `sandbox_mode = "read-only"`：默认禁止直接写文件
- `approval_policy = "on-request"`：需要越过限制时进入审批

这两项已经足够拦住大部分会改变项目状态的高风险动作。

## v2 的工作方式

`v2` 不是完全放开，也不是回到高频确认模式，而是采用下面这套原则：

- 在执行任何动作之前，先说明要做什么、为什么做
- 普通查看、分析、整理、读取等动作，说明后可以直接进行
- 遇到硬边界时，由 `read-only` 和 `on-request` 接管
- 遇到目标不清、信息不足、存在分支决策时，再停下来问用户

也就是说，`v2` 仍然强调“先讲明白意图”，但不再把“每一步都等用户确认”当成默认流程。

## 详细规则

更完整的 `v2` 规则写在 [`AGENTS.v2.md`](./AGENTS.v2.md)。

在 `v2` 里，这个文件更适合承担补充性约束，例如：

- 回复使用中文
- 不要擅自执行 git 操作
- Git 相关说明要非常详细，明确表明意图，并符合 Linus 对清晰、具体、可审查提交说明的要求
- 不确定时先问用户
- 执行动作前先说明意图

它不应该继续重复承担已经由运行时硬限制负责的审批职能。

## 这套限制如何生效

`v2` 的实际效果来自两层配合：

- 运行时硬限制负责拦住真正危险或越权的动作
- `AGENTS.v2.md` 负责约束沟通方式、行为边界和说明义务

因此，`v2` 的目标不是让 Agent 沉默执行，而是让 Agent 先说清楚，再在硬边界允许的范围内推进任务。

## v2 要解决的问题

`v1` 的主要问题不是限制不够，而是软限制和硬限制叠加后，容易出现这些情况：

- 很多动作明明已经会被 sandbox 拦住，还要先额外确认一轮
- 很多只读动作也被纳入重复确认流程
- 一次任务里经常出现多轮打断，推进效率很低

`v2` 就是为了解决这种“重复确认”问题。

它保留解释义务，保留硬边界，减少多余审批。

## 适用场景

`v2` 适合这样的使用方式：

- 希望 Agent 在行动前说明意图
- 希望高风险动作仍然被严格拦截
- 不希望普通任务推进时反复卡在“是否确认”上

它和 `v1` 的差别，不是更随意，而是把限制放在更合适的位置。
