# Agent 工作规则（v2）

## 修改规则

- 在修改任何文件之前，必须先说明准备修改什么、为什么修改、涉及哪些文件。
- 不要求每次普通动作都额外等待用户确认。
- 不要因为自己认为方案正确就直接改代码、配置或文档。
- 如果有多个可选方案，先列出差异和影响，让用户决定。
- 遇到目标不清、信息不足、存在明显分支决策时，必须先问用户，不要自行假设。

## 命令规则

- 执行任何命令之前，必须先说明命令用途和预期效果。
- 普通只读查看、分析、整理类命令，在说明用途后可以直接执行，不需要额外等待确认。
- 对于测试、运行脚本、生成文件、安装依赖等可能改变项目状态的操作，必须先说明用途；如果需要额外权限，再进入审批。
- 不要为了形式上的谨慎，重复要求用户确认那些本就会被系统限制拦住的动作。

## 学术引用规则

- 处理、新增或修改 BibTeX/.bib 引用时，必须对每一条目联网核实其真实性与字段准确性；未逐条核实不得作为可信引用使用，否则视为学术不端风险。
- 处理任何学术引用、数据、表格、事实陈述或结论性内容时，绝对不得相信模型自身记忆或未经核实的既有知识，必须通过联网检索（web search）核实可靠来源后再使用。
- 所有数据必须有明确、可核实的来源，不得编造、猜测或随意生成数据。
- 不得使用未经说明来源的 mock 数据、占位数据或自认为合理的数据替代真实数据；如确需示例数据，必须明确标注为示例且不得用于学术结论。
- 处理、生成、修改或引用任何数据表文件时，必须核实数据来源、字段含义、单位、时间范围和适用条件，不得凭经验或假设自行解释。
- 对无法核实来源或真实性的数据，必须明确说明不可信或待核实，不得作为学术依据使用。
- 严禁任何数据幻觉、引用幻觉、来源幻觉或未经核实的结论性表述；此类行为均视为学术不端风险。
- 任何可能构成学术不端的数据、引用、来源或结论问题，均按重大安全事故处理；必须立即停止相关生成或修改，并向用户明确说明风险、待核实事项和所需来源。
- 所有涉及数据、表格、统计结果、实验结果和文献依据的内容，必须保持学术端正、可追溯、可核查。

## Git 规则

- 除非用户明确要求，不要执行任何 git 操作。
- 用户要求 git 操作时，必须用中文说明每一步在做什么。
- commit 信息必须使用中文，并清楚描述本次变更内容。
- 第一次 commit 不要使用“初始化 xxx”“initial commit”这类空泛标题。
- commit 信息必须使用 Markdown 风格的多行正文，写清楚所有主要改动，不能只有一行。
- Git 相关说明和 commit 信息必须写得非常详细，明确表明意图，并符合 Linus 对清晰、具体、可审查提交说明的要求。
- commit 信息建议格式：

```text
完成 mmLock 毫米波雷达安全项目整理

- 整理论文 PDF、引用信息和来源说明
- 添加 radar FFT cube 处理 notebook
- 添加并行点云生成 pipeline
- 添加 CNN + BiLSTM 点云序列识别 notebook
- 整理论文图片和中文技术解读
```

- 不要主动执行 commit、push、pull、merge、rebase、reset、checkout 等操作。
- 不要回滚用户已有修改，除非用户明确要求。

## 沟通规则

- 回复使用中文。
- 操作前说明计划，操作后说明结果。
- 在执行任何动作之前，先讲明白意图，但普通动作不默认进入“等待确认”流程。
- 遇到不确定的地方，先问用户，不要自行假设后修改。
- 对于预计会持续较长时间的任务，优先使用 `tmux` 在后台运行，避免前台阻塞或中途中断。
