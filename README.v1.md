# codex-config

这个仓库用于记录我给 Codex/Agent 使用的限制方案。

## v1 限制方案

当前这套限制，本质上都属于 `v1`。

### 配置文件写法

如果希望以后每次在终端输入 `codex` 都默认启用这类限制，可以在本地全局配置文件 `~/.codex/config.toml` 中加入：

```toml
sandbox_mode = "read-only"
approval_policy = "on-request"
```

## 详细规则

更完整的 `v1` 规则写在 [`AGENTS.v1.md`](./AGENTS.v1.md)。

这个文件负责约束 Agent 的行为，例如：

- 修改前先说明准备改什么
- 必须等待用户确认
- 不要擅自执行 git 操作
- 回复使用中文

## 这套限制如何生效

这套限制不是只靠一个文件，而是多层叠加：

- `sandbox_mode = "read-only"` 会直接禁止写文件
- `approval_policy = "on-request"` 会让很多可能改变状态的动作进入审批
- [`AGENTS.v1.md`](./AGENTS.v1.md) 还会进一步要求先说明、先确认、不要擅改、不要擅自执行

叠加后的实际效果，就是 Agent 会被强行拉回“先提议、再等你点头”的工作模式。

## 为什么说它太强了

这套 `v1` 限制能有效防止 Agent 自作主张，但副作用也很明显：

- 很多本来正常的开发动作也会被拦住
- 做项目时经常要连续申请 permission
- 一次任务里可能要确认很多轮
- 在某些场景下，交互体验会像反复卡在确认流里，甚至看起来像死循环

所以它更适合“强控制、低自治”的使用方式，不太适合需要连续执行、频繁修改、反复测试的项目型工作流。
